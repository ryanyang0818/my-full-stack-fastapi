Dodo Admin 權限模型：角色導向 RBAC 與 Menu scope

本文件用權限議題示範 Dodo Admin 的底層架構文件格式。重點是把資料庫邏輯、前端頁面邏輯、程式碼邏輯拆開，避免把「選單顯示」誤認成真正權限。

摘要

  • Dodo Admin 採用 角色導向 RBAC：先定義使用者角色，再由角色取得功能權限。
  • Menu 是功能授權單位；有某個 menu_key，代表可以使用該功能範圍。
  • user_menu_override 提供個人例外，可針對單一使用者做 allow 或 deny。
  • 前端 Sidebar 只顯示使用者可見功能；真正權限 enforcement 必須在後端 require_menu(menu_key) 完成。
  • 目前不做細粒度權限，例如 CRUD 分權、欄位級權限、資料列級 owner_id 隔離。

相關的資料庫物件

資料庫物件 分類 用途
user 主體 代表登入者，是權限計算的起點。
role 角色 定義一群人的職責，例如 sales_staff、warehouse_manager。
user_role 關聯表 指定使用者擁有哪些角色。
menu 功能範圍 代表功能入口，也是 API 授權使用的 scope。
role_menu 授權表 指定角色可以使用哪些 menu 功能。
user_menu_override 例外表 針對單一使用者補充或排除 menu 權限。
view_user_visible_menu View 計算角色授權與個人例外後的最終可見 menu。

邏輯

資料庫邏輯

資料庫負責保存角色、選單與使用者例外，並透過 view 統一產生使用者最後可用的 menu 清單。

前端頁面邏輯

前端不自行推導權限，而是根據後端回傳的 menu tree 顯示入口；使用者點擊可見 menu 後，開啟對應 tab 或頁面。

程式碼邏輯

API controller 在進入業務邏輯前呼叫 require_menu(menu_key)。通過才執行查詢或 mutation，不通過直接回 403。

細節

規則 說明
角色是主要權限來源 大部分權限應透過角色分配，不應逐一為使用者客製。
Menu 是功能授權單位 有 inventory.products，代表可以進入商品管理功能。
Override 是例外機制 只用來處理少數個人例外，不應取代角色設計。
Deny 優先 若角色允許，但使用者被 deny，最終應視為無權限。
後端是最後防線 API 必須檢查 require_menu，不能只靠前端隱藏選單。

範例

角色授權

warehouse_staff 透過 role_menu 取得 inventory.products。

結果：使用者看得到商品管理選單，且 API require_menu("inventory.products") 通過。

個人 Allow

某使用者角色本來沒有 sales.orders，但被個別設定 allow。

結果：該使用者可以看到銷售訂單功能，也可以呼叫對應 API。

個人 Deny

某使用者角色本來有 inventory.stock，但被個別設定 deny。

結果：最終看不到該選單，直接呼叫 API 也應回 403。

跨域阻擋

sales_staff 直接呼叫 /inventory/products。

結果：即使知道 URL，也因缺少 menu key 而回 403。

React Component

重點檔案 簡短用途
frontend/src/components/Sidebar/AppSidebar.tsx 呼叫 MenusService.readMyMenuTree()，取得目前使用者可見的 menu tree。
frontend/src/components/Sidebar/TreeMenu.tsx 渲染後端回傳的樹狀選單，點擊 leaf 後開啟對應 tab。
frontend/src/components/Sidebar/tabKeyMap.ts 把 menu path 映射到前端 tab key，例如 /system/menus → menus。
frontend/src/components/TabHeader/tabRegistry.tsx 定義 tab key 對應的標題、icon 與實際 React component。
frontend/src/hooks/useAuth.ts 取得目前登入使用者資訊，供前端做顯示與基本狀態判斷。

FastAPI Controller

重點檔案 簡短用途
backend/app/api/deps.py 提供 CurrentUser、登入狀態解析與 superuser dependency。
backend/app/api/routes/menus.py 提供 /menus/tree 與 menu 管理 API，是 menu 權限資料的主要出口。
backend/app/models.py 定義 Role、Menu、RoleMenu、UserMenuOverride 等資料模型。
backend/app/alembic/versions/*role_menu* 建立 menu、role_menu 與角色選單相關 view。
backend/app/alembic/versions/*user_menu_override* 建立使用者個別 override 與 view_user_visible_menu。

避免誤會與不做的地方

不做細粒度權限

  • 不做 read / create / update / delete 分開控管。
  • 不做欄位級權限。
  • 不做資料列級 owner_id 隔離。

不把前端當權限來源

  • 隱藏按鈕只是 UX，不是安全。
  • 隱藏選單只是結果，不是 enforcement。
  • 直接打 API 必須仍然被後端擋住。
