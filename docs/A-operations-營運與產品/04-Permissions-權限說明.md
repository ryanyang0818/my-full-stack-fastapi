# Dodo Admin 權限說明

Dodo Admin 的權限設計，
可以分成兩個層次：

1. 先確認使用者是誰
2. 再確認使用者能使用哪些功能

登入負責確認身份。

權限負責決定功能範圍。

這兩件事必須分開理解，
不能把「已登入」誤認成「所有功能都可以使用」。

## 權限設計重點

| 項目 | 說明 |
|---|---|
| 身份確認 | 使用者必須先登入，系統才會承認他的身份 |
| 功能權限 | 不同角色會看到不同功能入口 |
| 選單範圍 | 選單代表使用者可進入的功能範圍 |
| 個人例外 | 特定使用者可以有額外開放或排除 |
| 後端檢查 | 真正的資料操作仍由後端確認權限 |

## 登入與身份確認

使用者進入 Dodo Admin 時，
必須先用帳號密碼登入。

登入成功後，
系統會發給使用者一個登入憑證。

之後使用者操作系統時，
每次請求都會帶著這個憑證，
讓後端確認「這個人是誰」。

如果憑證失效、帳號停用，
或使用者沒有登入，
系統就不會讓他繼續使用受保護的功能。

## 角色與功能權限

Dodo Admin 採用角色導向的權限設計。

也就是先定義角色，
再決定這個角色可以使用哪些功能。

例如：

| 角色 | 可使用功能 |
|---|---|
| 管理者 | 使用者管理、選單管理、系統設定 |
| 一般使用者 | 與自己工作相關的功能 |
| 特定職務 | 依職務開放對應功能 |

這樣做的好處是：

- 權限不用逐一設定在每個人身上
- 多數使用者可以透過角色快速管理
- 少數特殊情況再用個人例外處理

## 選單與權限的關係

在 Dodo Admin 中，
選單不只是畫面上的功能入口。

選單也代表一個功能範圍。

使用者能不能看到某個功能，
會先由後端依照角色與個人例外計算。

前端再依照後端回傳的結果，
顯示使用者可以看到的選單。

## 個人例外

大多數權限應該由角色管理。

但有些情況需要針對個人調整。

例如：

| 情境 | 說明 |
|---|---|
| 額外開放 | 某位使用者需要臨時看到特定功能 |
| 特別排除 | 某位使用者雖屬於角色，但不應看到某功能 |
| 職務過渡 | 使用者正在交接職務，需要短期調整功能 |

個人例外應該是少數情況，
不應取代角色設計。

## 表單與資料操作

表單、資料表與功能頁面，
都只是使用者操作資料的入口。

前端可以控制畫面顯示，
例如隱藏某個選單、按鈕或表單欄位。

但真正的資料操作，
仍必須由後端確認使用者是否有權限。

這代表：

- 看不到功能入口，是一種使用體驗控制
- 直接呼叫 API，也必須被後端檢查
- 資料新增、修改、刪除，都應經過後端權限規則

## 權限流程

| 流程 | 系統行為 |
|---|---|
| 使用者登入 | 系統確認身份並發出登入憑證 |
| 使用者進入後台 | 後端依角色與例外計算可見功能 |
| 前端顯示選單 | 只顯示使用者可使用的功能入口 |
| 使用者操作資料 | 後端再次確認身份與權限 |
| 權限不足 | 系統拒絕操作，不執行資料變更 |

## 安全原則

| 原則 | 說明 |
|---|---|
| 身份與權限分離 | 登入只代表身份成立，不代表擁有所有權限 |
| 後端是權限依據 | 前端顯示結果不能取代後端檢查 |
| 角色優先 | 大多數權限應透過角色管理 |
| 例外克制使用 | 個人例外只處理少數特殊情況 |
| 拒絕優先 | 若同時有開放與排除，應以排除為準 |

## 核心範圍

這份權限說明聚焦在 Dodo Admin 的基礎權限能力：

- 登入身份確認
- 角色導向權限
- 選單功能範圍
- 個人例外
- 後端權限檢查

更細緻的權限能力，
例如欄位級權限、資料列級權限、
或每個按鈕的獨立權限，
可以視未來實際業務需求再擴充。
