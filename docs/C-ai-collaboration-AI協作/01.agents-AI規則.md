# Dodo Admin AI 協作規則

本文件整理目前在 Dodo Admin 專案中，
Ryan 與 AI agent 協作時使用的規則、模型分工、
skill 安裝位置與相關資料夾。

範圍限定在本專案可確認的設定：

- `AGENTS.md`
- `CLAUDE.md`
- `.codex/`
- `.claude/`
- `.agents/skills/`
- `skills-lock.json`

## 01. 協作定位

Dodo Admin 的 AI 協作不是單純讓 AI 寫程式。

目前比較像是：

```text
Ryan
  ↓
提出產品 / 文件 / 工程方向
  ↓
AI agent
  ↓
讀取專案規則與既有檔案
  ↓
提出計畫或直接執行
  ↓
產出程式碼、文件、測試、commit 建議
```

AI agent 的主要責任：

| 責任 | 說明 |
|---|---|
| 文件整理 | 將討論內容轉成可保存的 Markdown / HTML |
| 程式修改 | 依照現有架構做最小範圍修改 |
| 架構梳理 | 將系統角色、資料流、權限流整理成可讀文件 |
| 測試協助 | 判斷應跑哪些後端或前端測試 |
| Git 協作 | 依 SOP 整理 staged files、commit message 與完成計畫 |
| 專案知識保存 | 將反覆討論出的規則沉澱成 docs 或 skill |

## 02. 目前使用的 AI 模型

目前專案中主要有兩種 AI model / AI 工具協作脈絡：

| AI | 主要用途 | 專案內入口 |
|---|---|---|
| Codex / GPT-5 | 目前主要協作代理，負責讀檔、改檔、整理文件、執行命令 | `AGENTS.md`、`.codex/`、`.agents/skills/` |
| Claude / Claude Code | 早期或平行使用的協作代理，透過 `CLAUDE.md` 接回共同規則 | `CLAUDE.md`、`.claude/` |

重點不是兩個模型彼此直接溝通，
而是它們共用同一套專案規則與文件。

```text
Ryan
  │
  ├─ Codex / GPT-5
  │    └─ reads AGENTS.md → .codex/README.md → project docs / skills
  │
  └─ Claude / Claude Code
       └─ reads CLAUDE.md → AGENTS.md → same project rules
```

因此協作核心是：

```text
不同 AI model
  ↓
共同讀取專案規則
  ↓
使用相同 docs / SOP / skills
  ↓
產出可被下一個 AI 接續的檔案
```

## 03. 共同規則入口

| 檔案 | 用途 |
|---|---|
| `AGENTS.md` | 專案主要 AI 協作規則入口 |
| `CLAUDE.md` | Claude 入口，要求遵循 `AGENTS.md` |
| `.codex/README.md` | Codex 文件索引與讀取順序說明 |
| `.codex/config.toml` | Codex 專案設定 |
| `.codex/environments/environment.toml` | Codex app actions，例如 `bun run dev`、`bun run test` |

目前 `CLAUDE.md` 很短，核心內容是：

```text
閱讀並遵循 AGENTS.md 及其引用鏈的所有規則。
```

這代表 Claude 的專案規則不另外分叉，
而是回到 `AGENTS.md`。

## 04. Codex 協作方式

Codex 目前在本專案的工作方式：

| 階段 | 做法 |
|---|---|
| 讀取規則 | 先依 `AGENTS.md` 與 `.codex/README.md` 理解專案規則 |
| 探索 | 使用 `rg`、`Get-Content`、`git status` 讀取真實檔案 |
| 計畫 | 重要開發任務先列計畫，必要時存入 `docs/執行計畫/` |
| 實作 | 使用 `apply_patch` 修改檔案 |
| 驗證 | 依任務選擇檔案檢查、測試或瀏覽器檢查 |
| Git | commit 前詢問測試與 message，依 SOP 處理 |
| 文件 | 將討論結果沉澱成 `docs/` 內可保存文件 |

Codex 不是只依賴聊天記憶，
而是盡量把重要決策寫入 repo 文件。

## 10. 協作流程與 SOP

目前實際合作大致是：

```text
Ryan 提出需求
  ↓
AI 讀取相關文件 / 程式碼
  ↓
必要時先提出計畫
  ↓
Ryan 確認方向
  ↓
AI 實作或產出文件
  ↓
AI 驗證結果
  ↓
Ryan 決定是否測試 / Git / 歸檔
```

若進入正式 SOP，流程會變成：

```text
探索 / 提出計畫
  ↓
存入計畫
  ↓
實作
  ↓
測試
  ↓
Git
  ↓
完成計畫歸檔
```

目前計畫保存位置：

```text
docs/執行計畫/
```

完成計畫位置：

```text
docs/執行計畫/完成計畫/
```

## 11. 協作原則

| 原則 | 說明 |
|---|---|
| 以 repo 文件為準 | 重要規則要沉澱到 `docs/`、`AGENTS.md` 或 skill |
| 先讀真實檔案 | 不憑印象推測專案結構 |
| 小步修改 | 優先保持修改範圍小 |
| 不覆蓋使用者改動 | 遇到既有異動要先辨識再處理 |
| 文件可接手 | 讓下一個 AI model 可以從文件接續工作 |
| skill 要專案化 | Dodo Admin 專用規則放 `.agents/skills/` |
| 測試與 Git 分開確認 | commit 前先詢問是否測試 |
