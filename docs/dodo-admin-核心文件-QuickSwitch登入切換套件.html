QuickSwitch Foundation · Version 1

Dodo Admin QuickSwitch：登入頁快速切換角色帳號的瀏覽器擴充套件

本文件說明 QuickSwitch 套件的運作機制，把「套件自己注入的 UI」「套件填值並送出表單的動作」「真正驗證帳密的後端邏輯」三層拆開，避免把「套件幫你填好表單」誤認成「套件繞過了登入驗證」——它從頭到尾都只是在自動化使用者本來會做的操作。

摘要

  • QuickSwitch 是 Manifest V3 的 content script，只在登入頁生效（`matches` 限定 `localhost:5173`~`5179` 七個 port 的 `/login*`）。
  • 11 組角色帳號（含密碼）明碼寫死在套件的 `accounts.js`，不經任何 API 取得，純粹是套件自己的設定資料。
  • 因為登入頁是 React controlled input，套件不能直接 `input.value = x`，必須用 native setter + `dispatchEvent` 才能讓 React Hook Form 偵測到變化。
  • 登入頁的表單是 SPA 非同步渲染出來的，套件用輪詢等表單出現（每 300ms 檢查一次，10 秒逾時放棄），不是假設頁面載入完成表單就已經存在。
  • 登入成功後 SPA 路由跳轉不會整頁刷新，套件靠另一個輪詢偵測網址變化，離開 `/login` 就移除注入的面板。
  • 套件完全不繞過後端驗證——選帳號之後做的事，就是真人會做的「打字、按登入」，最終一樣要走 `/login/access-token` 的 `authenticate()` 檢查。

相關物件（套件設定與資料來源）

這個子系統不是網站本身的功能，沒有資料庫，所以這裡列的是套件運作時依賴的設定、資料與目標頁面契約。

物件 分類 用途
manifest.json content_scripts.matches 設定 決定套件在哪些網址生效，目前是逐一列出 7 個 localhost port 的 `/login*`。
window.__DODO_ACCOUNTS__（`accounts.js`） 資料 11 組 `{label, email, password}`，是下拉選單的資料來源，密碼明碼存放。
[data-testid="email-input"] / [data-testid="password-input"] 目標選擇器 套件依賴登入頁這兩個屬性存在，才能找到要填值的欄位。
form button[type="submit"] 目標選擇器 填完值後，套件找這顆按鈕並模擬點擊，觸發真正的登入送出。
#dodo-quickswitch-panel 套件自產 DOM 套件自己注入的浮動面板，不是登入頁原生的元素，離開登入頁時要主動移除。

細節

規則 說明
不能直接 `input.value = x` React controlled input 不會偵測到原生賦值，必須用 `Object.getOwnPropertyDescriptor` 拿 native setter 再 `dispatchEvent("input")`。
`document_idle` 不保證表單已渲染 React 是 SPA，DOM ready 跟「React 把表單畫出來」是兩個不同時間點，套件用每 300ms 輪詢、10 秒逾時的方式等待，不是賭時機。
SPA 路由跳轉不會重新注入 content script 登入成功後網址變了但沒有整頁刷新，套件不會自動感知，要另外輪詢 `location.pathname` 才能在離開登入頁時移除面板。
Manifest V3 不支援 port 範圍語法 `matches` 只能逐一列出 5173~5179 七個 port；如果 dev server 意外跑在範圍外的 port，套件不會生效，也不會有任何提示。

範例

正常切換帳號

在 `localhost:5173/login` 選擇「業務主管」。

結果：email/password 自動填入並送出，成功登入，依角色看到對應選單。

登入頁 DOM 結構被改名

未來若 `data-testid="email-input"` 被移除或改名。

結果：套件找不到欄位，輪詢 10 秒後放棄，不報錯、不影響頁面，只是悄悄失效。

dev server 跑在範圍外的 port

假設 dev server 意外啟動在 `localhost:5180`。

結果：`matches` 沒涵蓋這個 port，content script 完全不會注入，畫面上不會出現任何提示。

登入成功後離開 /login

選帳號送出登入，SPA 路由跳轉到 `/`。

結果：套件偵測到 `pathname` 不再是 `/login`，自動移除浮動面板，不會殘留在後台畫面上。

套件檔案與依賴的目標檔案

重點檔案 簡短用途
extensions/dodo-quickswitch/manifest.json Manifest V3 設定，宣告 content script 的生效網址與載入順序。
extensions/dodo-quickswitch/accounts.js 11 組角色帳號明碼清單，掛在 `window.__DODO_ACCOUNTS__`。
extensions/dodo-quickswitch/content.js 注入面板、輪詢偵測表單與路由、安全填值並自動送出的全部邏輯。
frontend/src/routes/login.tsx 套件依賴的目標頁面，提供 `data-testid="email-input"`/`password-input` 與 submit 按鈕；這支檔案的結構是套件能正常運作的前提。

套件最終驅動到的後端邏輯

套件本身不呼叫任何後端 API，它只是自動化「使用者輸入 + 按下登入」這個動作；真正的驗證仍是下面這條既有流程在跑。

重點檔案 簡短用途
backend/app/api/routes/login.py `/login/access-token`，套件按下送出後，前端真正打的 API，跟使用者手動登入完全相同。
backend/app/crud.py `authenticate()` 仍會對套件填入的帳密做完整 Argon2 比對與 timing attack 防護，套件不享有任何特殊通道。

避免誤會與不做的地方

不做的事

  • 不做帳密加密儲存，密碼明碼寫在 `accounts.js`，假設本機開發環境是安全的。
  • 不繞過後端任何驗證邏輯，包括前端 zod schema 與後端 `authenticate()`。
  • 不支援正式環境網域，`matches` 只寫死本機開發用的 7 個 port。
  • 不上架 Chrome Web Store，僅以「載入未封裝項目」方式安裝。

不把套件自身的時機判斷當成即時保證

  • 表單偵測與路由偵測都是輪詢，最高有 300ms 延遲，不是即時反應。
  • 套件依賴 `data-testid` 選擇器存在；這兩個屬性若被移除，套件會靜默失效，不會主動通知任何人。
  • `matches` 是寫死的 port 清單，dev server 換成清單外的 port，套件就完全不會出現。
