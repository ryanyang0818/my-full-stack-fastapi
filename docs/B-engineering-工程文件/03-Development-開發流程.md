# Dodo Admin 開發流程

本文件先保留開發流程的章節骨架。

它不是要重複 `技術棧` 或 `架構`，
而是用來回答：

> 工程師或 AI agent 要怎麼把一個功能，
> 正確地做進 Dodo Admin 這套系統？

目前 `CRUD` 與 `Master-Detail` 尚未完全討論清楚，
因此本版先不展開細節。
後續等這兩個核心方向確認後，
再補成可執行的開發 SOP。

## 01. 開發流程總覽

說明一個功能從想法、規格、資料模型、API、
前端畫面、權限、測試到文件同步的完整路徑。

```text
Idea
  ↓
Spec
  ↓
Model / API / UI
  ↓
Permission
  ↓
Tests
  ↓
Docs
```

## 02. 新功能開發流程

說明新增一個功能時，
應該先確認哪些問題：

- 功能目的
- 使用者入口
- 資料來源
- API 邊界
- 權限邊界
- 測試方式
- 文件同步範圍

## 03. API 契約更新流程

說明前後端 API 如何同步。

重點包含：

- 後端 route 與 schema 變更
- OpenAPI 規格更新
- 前端 API client 重新產生
- 前端呼叫端調整
- 型別錯誤與測試驗證

## 04. 權限功能開發流程

說明新功能如何接入權限系統。

重點包含：

- menu key 是否需要新增
- 角色權限是否需要設定
- 使用者覆寫是否受影響
- Sidebar 是否顯示
- 後端 API 是否需要權限保護

## 05. UI 元件開發流程

說明新畫面或新元件如何接進前端系統。

重點包含：

- 是否能沿用既有元件
- 是否需要抽成共用元件
- 是否進入 tab 工作區
- 是否需要 side panel、dialog、table、form
- 是否符合目前設計系統

## 06. 資料庫變更流程

說明新資料表、新欄位或資料關係變更時，
應該如何處理。

重點包含：

- model 調整
- migration 建立
- seed data 是否需要更新
- 舊資料相容性
- 測試資料與本機 DB 驗證

## 07. 測試補強流程

說明功能完成後，
應該補哪些測試。

重點包含：

- 後端 API 測試
- 資料庫與權限測試
- 前端元件或互動測試
- Playwright 端對端測試
- 回歸測試範圍

## 08. Bug 修復流程

說明修復問題時，
應該先定位根因，
再做最小範圍修正。

重點包含：

- 重現問題
- 找出共用入口或 shared function
- 修正根因而不是症狀
- 補上最小可驗證測試
- 記錄影響範圍

## 09. 文件同步流程

說明功能完成後，
哪些文件可能需要同步。

重點包含：

- 工程文件
- 營運與產品文件
- 權限說明
- API 或資料流文件
- AI agent 協作規則
- todo / 執行計畫歸檔

## 10. AI 協作開發流程

說明 AI agent 接手開發任務時，
應該如何讀取上下文與控制修改範圍。

重點包含：

- 先讀需求與相關文件
- 先確認架構角色與資料流
- 先搜尋既有實作
- 優先沿用既有模式
- 避免不必要抽象
- 修改後回報影響檔案與驗證狀態
