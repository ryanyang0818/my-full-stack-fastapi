Dodo Admin 系統登入：帳密驗證、JWT 與密碼復原

本文件說明 Dodo Admin 目前的登入機制，把資料庫邏輯、前端頁面邏輯、程式碼邏輯拆開，避免把「前端存有 token」誤認成「後端已驗證身份」。

摘要

  • 登入採用 OAuth2 Password Flow：帳號密碼換取一張 JWT access_token，沒有 refresh token、沒有 session 表。
  • 密碼用 Argon2（備援 Bcrypt）雜湊，原文密碼從不落地、也不會出現在任何 log。
  • authenticate() 對「帳號不存在」與「密碼錯誤」一律回應相同錯誤，並用 dummy hash 補一次驗證運算，防 timing attack 與帳號列舉。
  • 前端把 access_token 存在 localStorage，之後每個 API 請求帶 Authorization: Bearer；token 過期或失效則導回登入頁。
  • 忘記密碼走 Email Token 流程：寄送有時效的重設連結，過程中同樣不洩漏「這個信箱是否存在」。
  • 目前不做 2FA / OTP、不做登入失敗鎖定、不做多裝置 session 管理。

相關的資料庫物件

資料庫物件 分類 用途
user.hashed_password 憑證 Argon2 雜湊後的密碼，是 authenticate() 比對的依據。
user.is_active 狀態 停用帳號即使密碼正確仍無法登入，登入與 token 驗證皆會檢查。
user.last_login_at 紀錄 每次 authenticate() 成功就更新，用於追蹤最後登入時間。
JWT access_token 憑證（非資料表） 不落地資料庫，內容是 sub=user.id + 到期時間，由 SECRET_KEY 簽章。
密碼重設 token 憑證（非資料表） 同樣是簽章後的 JWT，不存資料庫，靠到期時間與簽章本身防偽造。
DUMMY_HASH 防護機制 固定的 Argon2 雜湊常數，帳號不存在時仍執行一次驗證運算，拉平回應時間。

邏輯

資料庫邏輯

資料庫只保存「密碼雜湊」與「帳號狀態」，不保存 token、不保存登入 session；身份驗證的結果只活在 JWT 裡。

前端頁面邏輯

前端不驗證密碼，只負責送出表單、保存 token、並在之後每個請求附帶它；token 失效時統一導回登入頁，不嘗試自行判斷身份是否有效。

程式碼邏輯

每個受保護的 API 都透過 get_current_user 解析 JWT、查回 User，並檢查 is_active；任何一步失敗即回 403/400，不會放行。

細節

規則 說明
密碼一律雜湊，不存明文 get_password_hash() 用 Argon2，舊資料若曾用 Bcrypt 仍可驗證並自動升級雜湊。
帳號不存在與密碼錯誤回應一致 避免攻擊者藉錯誤訊息差異枚舉出哪些 email 已註冊。
Timing attack 防護 帳號不存在時仍對 DUMMY_HASH 執行一次驗證，讓回應時間不洩漏「帳號是否存在」。
token 是唯一憑證 沒有 server-side session，登出只是前端刪除 localStorage 裡的 token，後端不會主動失效它（到期前仍可用）。
停用帳號隨時擋下 is_active=false 不論登入或既有 token 驗證，都會被擋，即使 token 尚未過期。
密碼重設不洩漏帳號存在與否 無論信箱是否註冊，/password-recovery/{email} 回應文字都相同。

範例

正常登入

帳密正確且 is_active=true。

結果：回傳 access_token，last_login_at 更新。

密碼錯誤

帳號存在，但密碼比對失敗。

結果：回 400 Incorrect email or password，跟帳號不存在時的訊息完全相同。

帳號被停用

帳密正確，但 is_active=false。

結果：回 400 Inactive user；若該使用者手上還有未過期的舊 token，呼叫任何 API 也會被 get_current_user 擋下。

忘記密碼

使用者送出信箱，無論是否註冊過。

結果：畫面回應一律是「若該信箱已註冊，我們已寄出重設連結」；只有真的註冊過才會收到信。

React Component

重點檔案 簡短用途
frontend/src/routes/login.tsx 登入頁路由與表單畫面，呼叫 useAuth().login() 送出帳密。
frontend/src/hooks/useAuth.ts 封裝 LoginService.loginAccessToken，登入成功寫入 localStorage，登出時清除並重置頁籤狀態。
frontend/src/client（自動產生 SDK） LoginService 對應 /login/access-token 等後端 API，型別隨 OpenAPI schema 重新產生。
frontend/src/components/Common/*（路由 guard） 讀取 isLoggedIn()（即 localStorage 是否有 token）決定是否導向登入頁。

FastAPI Controller

重點檔案 簡短用途
backend/app/api/routes/login.py 提供 /login/access-token、/login/test-token、/password-recovery、/reset-password。
backend/app/crud.py authenticate() 與 DUMMY_HASH，是帳密驗證與防 timing attack 的核心邏輯。
backend/app/core/security.py create_access_token、verify_password、get_password_hash，封裝 JWT 與 Argon2/Bcrypt。
backend/app/api/deps.py get_current_user 解析 token 並查回 User，是所有受保護 API 的共用入口。
backend/app/utils.py generate_password_reset_token、verify_password_reset_token、寄信內容組裝。

避免誤會與不做的地方

不做的安全機制

  • 不做登入失敗次數鎖定（目前無暴力破解防護）。
  • 不做 2FA / OTP（帳密驗證後即直接核發 token）。
  • 不做圖形驗證碼（CAPTCHA），尚未串接第三方服務。

不把前端當身份來源

  • localStorage 有 token 只代表「曾經登入過」，不代表後端一定承認。
  • token 過期或帳號被停用後，前端要靠 API 回應的 401/403 才會知道並登出。
  • 沒有 server-side session，後端無法「主動踢人下線」，只能等 token 到期。
