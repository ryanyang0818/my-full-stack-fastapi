# Product

## Register

brand

## Users

主要對象是公司內部**營運/管理層**——他們不需要理解技術實作,但需要清楚知道 Dodo Admin 是什麼、不是什麼,以及它的邊界在哪裡。

次要對象:
- 新加入的工程師與 AI agent,用來快速理解產品定位、降低交接成本
- 潛在客戶/合作對象,用於對外展示產品定位

閱讀情境:多半是抱著「這是什麼系統?能不能信任它?」的疑問點進來,不是要來查技術文件。

## Product Purpose

一個獨立於現有 admin 前端之外的**單頁式產品定位介紹網站**,把 `docs/A-operations-營運與產品/` 底下的 5 份核心文件(產品意圖、功能規格、使用流程、權限說明、常見問題)轉譯成一則連貫的敘事,而非條列式技術文件。

核心任務:讓讀者在不需要理解任何程式碼的前提下,清楚掌握「Dodo Admin 是企業後台管理系統的基礎骨架,不是完整 ERP」這個定位,以及它明確劃出的功能邊界。

成功的樣子:讀完網站的營運人員,能自己向別人複述「Dodo Admin 做的是地基,不是蓋好的城市」這個核心比喻,並理解身份確認與功能權限分離的安全原則。

## Brand Personality

**冷靜・結構化・安全**

- 冷靜:深藍/深青為主色,呼應「信任與安全」而非熱情行銷的語氣
- 結構化:淺灰背景 + 白色內容區塊 + 淺灰藍分隔線,版面像儀表板一樣條理分明
- 安全:金色/綠色驗證色只用於「通過檢查、安全鎖定」等明確語意,不做裝飾用途

視覺語言改採儀表板(dashboard)風格的結構化視覺元素——圖示、狀態標記、結構化區塊——取代先前規劃的插畫敘事路線(工頭/房子故事線已放棄,不再使用)。內容定位仍是行銷/介紹頁(brand register),但視覺語言刻意向 product register 的儀表板美學靠攏,是一次刻意的方向決策,而非未定案。

## Anti-references

- 不要像典型行銷頁一樣誠容化:拒絕 hero-metric 樣板(大數字+小標籤+漸層強調)、拒絕漸層文字、拒絕千篇一律的圖示+標題卡片網格
- 不要用流程圖呈現概念性內容——結構化區塊/狀態標記可以,但不要把定位性內容畫成技術流程圖
- 不要讓「功能規格」這類技術性內容搶走「產品意圖」「權限說明」等定位性內容的篇幅與版面份量

## Design Principles

1. **地基先於裝飾**——先把定位與邊界講清楚,視覺服務內容,不喧賓奪主
2. **結構化視覺語言貫穿全站**——深藍主色、淺灰背景、白色區塊、淺灰藍分隔線的儀表板美學,作為五個段落共用的視覺骨幹
3. **克制勝於誇飾**——避免典型行銷頁的誇大用語與模板化結構(hero-metric 等),語氣沉穩篤定
4. **依重要性分配篇幅**——定位與邊界類段落(產品意圖、權限說明)篇幅較大;技術性段落(功能規格)刻意收斂
5. **敘事誠實對應產品原則**——像產品本身「拒絕優先」的安全原則一樣,網站也誠實標示邊界,不過度承諾

## Accessibility & Inclusion

基本 WCAG AA 即可,無特殊額外需求。需留意驗證色(金/綠)標記與圖示不能只靠顏色傳達語意,須搭配文字或圖形樣式,確保色盲使用者也能辨識「已驗證/已通過」狀態。
