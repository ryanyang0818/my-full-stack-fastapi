Product boundary consolidation

Dodo Admin 的邊界：本體是後台骨架，價值來自落地能力。

這份頁面收攏目前討論結果：Dodo Admin 不把自己定義為完整 ERP，也不走 No Code 表單設計器路線。 它要呈現的是高度客製的畫面，以及可重複使用的後台開發模式。

核心結論：畫面要像客製系統一樣貼近情境，底層要像產品一樣可重複、可維護、可擴充。

Current decisions

目前已經定下來的產品邊界。

這些不是最終 PRD，而是接下來避免範圍失控的工作假設。

決策 1

選 C，不選 B

Dodo Admin 本體是通用後台骨架；ERP 只是 Demo / 樣板，不是產品主身份。

決策 2

Forms 選 C

不做拖拉式表單設計器，也不把所有畫面限制在設定檔生成。Demo 畫面可以客製。

決策 3

標準化在底層

單表 CRUD、表格、表單送出、驗證、權限、資料溝通可以沉澱成模式。

Boundary 01

Dodo Admin 不承諾完整 ERP 產業邏輯。

這是重要風險控管：目前真正掌握的是後台系統落地能力，不是庫存、採購、會計、BOM、排程等完整 ERP 知識。

選 B 會變成什麼？

  • 產品本體會變成一套小型 ERP。
  • 客戶會期待採購、庫存、工單、出貨、會計等完整流程。
  • 產品會被特定產業知識拖住，開發與顧問壓力快速升高。

選 C 的正確定義

  • Dodo Admin 本體是後台骨架。
  • ERP Demo 用來展示小型製造 / 傳產情境。
  • Demo 可以變成樣板，但不代表承諾完整 ERP 解決方案。
Dodo Admin 不提供完整 ERP 解決方案，而是提供一套可以支撐 ERP-like 情境的後台骨架。

Boundary 02

Forms 不走 No Code，也不把畫面綁死在設定檔。

你要呈現的是系統很貼近場景、畫面很有客製感；如果所有畫面都由設定檔產生，表現力會被限制。

選項 做法 目前判斷
A：No Code 表單設計器 拖拉欄位、即時產生表單。 第一階段不做，範圍太大，容易變成平台競賽。
B：設定檔驅動所有表單 用 config 自動產生表單 UI。 暫不主打，畫面容易受到限制。
C：客製畫面 + 可沉澱模式 畫面依 Demo / 客戶情境客製，底層模式共用。 目前選擇。

Development model

開發邏輯：畫面客製，模式重用。

Dodo Admin 不是賣「不用工程師」，而是賣「工程師可以更快落地」。
