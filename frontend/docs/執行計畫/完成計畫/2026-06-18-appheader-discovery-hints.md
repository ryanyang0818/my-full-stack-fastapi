# AppHeader 可發現性提示計畫

## 目標

讓使用者更容易發現「滑鼠移到螢幕上緣可以開啟 AppMenuBar」這個互動，並用小步實作逐一確認最終方向。

## 背景

目前 AppMenuBar 預設隱藏，使用者需要把滑鼠移到螢幕最上緣才會觸發。這個互動對熟悉桌面軟體的人合理，但對第一次使用的人 discoverability 偏低。

## 候選方案

| 方案 | 名稱 | 做法 | 優點 | 風險 |
|---|---|---|---|---|
| A | 上緣細線提示 | 在畫面最上方保留 1-2px 感應線 | 最低干擾，保留乾淨畫面 | 新手可能仍看不懂 |
| B | Hover 小箭頭 | 滑到頂部感應區時，先出現向下箭頭或陰影 | 比細線更明確 | 動畫若太強會干擾 |
| C | 首次 Tooltip | 第一次進系統提示「滑過上緣開啟功能選單」 | 對新手最清楚 | 需要 dismiss / 已讀狀態 |
| D | 固定小把手 | 頂部中央放小 pill / grip / `⌄` | 可發現性高，仍低干擾 | 會永久佔一點視覺注意力 |
| E | 顯式按鈕入口 | 在 UserHeader 加一個小圖示按鈕開啟 AppMenuBar | 最穩、可鍵盤操作 | 和隱藏式互動重複 |

## 建議方向

先從低干擾、高可發現性的組合開始：

1. 固定小把手。
2. Hover 小把手時展開 AppMenuBar。
3. 外部點擊關閉。
4. 首次 Tooltip 作為可選增強。

## 一次展示多方案的做法

可以先做一個「開發用 variant switcher」，不用一次決定最終設計。

| 做法 | 說明 |
|---|---|
| `const discoveryVariant = "grip"` | 先用常數切換，不接 UI |
| `line / arrow / tooltip / grip / button` | 每次改常數即可看不同方案 |
| 不加正式設定頁 | 避免為暫時比較引入多餘架構 |
| 最終只保留勝出方案 | 選定後刪掉其他 variant code |

## 預計影響檔案

| 檔案 | 動作 | 說明 |

## 不做範圍

| 不做 | 原因 |
|---|---|
| 不一次導入正式偏好設定 | 目前只是互動方案評估 |
| 不使用大面積教學遮罩 | ERP 介面應維持安靜、低干擾 |
| 不改 Menubar dropdown 內容 | 本次只處理發現入口 |

## 驗收方式

- 使用者不用記憶隱藏互動，也能看出螢幕上緣可互動。
- 提示不遮擋 Topbar、Sidebar 或主要內容。
- Hover 展開與外部點擊關閉仍維持正常。
- 手機版不出現不必要的桌面 hover 提示。
