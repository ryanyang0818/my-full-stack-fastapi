# 頁籤系統設計決策重點

> 本文件彙整本次討論定案的頁籤(Tab)系統設計。**純設計決策,尚未實作。**

## 一、名詞約定
- 「頁面」這個模糊詞退休,一律改用專屬名稱。
- **路由頁** = Router 驅動、會改網址的傳統頁。
- **頁籤 / Tab** = 放在 Tab 容器裡渲染的元件,狀態驅動、不碰 Router。
- **頁籤標籤 / tab chip** = 頁籤列上可點可關的那一條。

## 二、互動
- **不去重**,一律可重複開啟(同功能可多開、各帶狀態)。
- 同名頁籤**不編號**(編號反而難管理)。
- 總量上限 **20**;第 21 個 → **硬擋**提示「請先關閉再開」,不自動關閉。
- 關閉 active 頁籤 → **右邊遞補**,沒右邊才往左。
- **第一個頁籤固定**:永久保留、無 X、不可關(內容待定:Dashboard 或 Index)。
- **右鍵選單三項**:① 關閉本頁籤 ② 關閉右側頁籤 ③ 關閉全部頁籤。
- **拖拉排序**:用 dnd-kit;**只能水平拖拉**;第一個固定頁籤完全鎖定(不可拖、不可移位,其他頁籤不能插到它前面,永遠最左)。
- **未存防呆**:不做。
- **顯示溢位**:交給套件處理,不特別設計。
- **釘選**:暫不討論。

## 三、架構
- 頁籤**脫離 Router**,直接渲染,由控制元件讀 store 決定輸出。
- 網址**路線 A:完全不碰網址**;上一頁/下一頁與頁籤無關。
- 唯一真相 = **Zustand store**:`{ openTabs:[{id,標題,元件,參數,狀態}], activeId }`。
- **Tab ID** 自動產生(不顯示於網址)。
- **路由頁分類**:
  - 既有頁 → 保留為路由頁。
  - 後續新頁 → 由使用者口頭指明是頁籤。
  - **Items → 轉頁籤**。

## 四、Items 轉頁籤(選項 A)
- **移除 Items 路由**,`/items` 失效(導回首頁)。
- Items 頁籤由**側邊選單點擊**開啟(呼叫 `store.openTab`),不靠網址。
- 內容抽成**純元件**(如 `ItemsView`)供頁籤 render。
- 舊的 `<Link to="/items">` / `navigate` 會被 **TS 標紅**,當待改清單逐一改成「開頁籤」動作。
- 「打 /items 會多開頁籤」的舊行為將**消失**(網址與頁籤脫鉤)。

## 五、TypeScript 考量
- 「路由頁」與「頁籤」兩種輸出**不會**害 TS 報錯(都是普通 React 元件)。
- 真正的 TS 風險來自**移除被引用的路由**(TanStack 型別化路由)→ 反而當免費 checklist。

## 六、生命週期
- 路線 **B**:只存「**輕量清單**」(開了哪些、順序、哪個 active);內容各自重載,內部狀態不留。
- 存放 = **localStorage**。
- **唯一重置點 = 登入成功時**:清空並寫入初始狀態(只剩固定頁籤)。
- **F5 重整**:init 只**讀** localStorage,不清、不判斷 → 保留舊頁籤。
- 結果:**登入 = 新的,F5 = 舊的**,只有一個控管點。
