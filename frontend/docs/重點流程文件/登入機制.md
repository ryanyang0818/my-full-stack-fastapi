# 登入機制

## 1. 整體流程

```
使用者輸入帳密 → loginMutation → LoginService.loginAccessToken (OAuth2 表單)
→ 後端回傳 access_token → 存入 localStorage → 導向 "/"
```

## 2. 核心檔案

| 檔案 | 角色 |
|---|---|
| `src/hooks/useAuth.ts` | 登入/登出/註冊邏輯核心 |
| `src/routes/login.tsx` | 登入頁面 UI + 表單驗證 |
| `src/main.tsx` | 全域 API 設定與錯誤處理 |
| `src/client/core/request.ts` | API 請求底層,附加 Authorization header |

## 3. 登入流程詳解

### 3.0 進入登入頁前的檢查(beforeLoad)

在使用者看到登入畫面之前,`login.tsx` 的 `beforeLoad` 會先檢查 `isLoggedIn()`(即 `localStorage.getItem("access_token") !== null`)。若 token 已存在,會立即 `throw redirect({ to: "/" })`,使用者完全不會看到登入表單,直接被導向首頁。**但這個檢查只判斷 token 是否「存在」,不驗證其是否仍有效或已過期**。

因此若 localStorage 中留有一個已過期的 token,流程會變成:`/login` → beforeLoad 判定已登入 → 導向 `/` → 頁面內的 API 請求(如 `currentUser`)回應 401 → `main.tsx` 的全域錯誤處理清除 token 並 `window.location.href = "/login"`(整頁刷新)。也就是使用者會經歷一次「短暫跳到首頁 → 又被踢回登入頁」的迴圈,而不是直接停留在登入頁面。這是與「6. 全域 401/403 處理」段落交互產生的邊界情境,目前專案沒有在進站時主動驗證 token 有效性(例如呼叫 `/login/test-token`)來避免此迴圈。

### 3.1 使用者輸入帳密

`login.tsx` 使用 `react-hook-form` 搭配 `zodResolver` 管理表單狀態,驗證規則定義在 `formSchema`。`username` 欄位驗證為 email 格式,`password` 至少 8 字元,驗證模式設為 `onBlur`(欄位失焦時觸發)。表單型別 `FormData` 透過 `satisfies z.ZodType<AccessToken>` 確保與後端 API 的 `Body_login_login_access_token` 型別一致。送出時呼叫 `form.handleSubmit(onSubmit)`,若驗證未通過則不會進入下一步。

### 3.2 loginMutation

`useAuth.ts` 使用 `@tanstack/react-query` 的 `useMutation` 包裝登入動作,提供 `isPending`、`onSuccess`、`onError` 等狀態管理。`onSubmit` 會檢查 `loginMutation.isPending`,避免使用者重複點擊送出造成多次請求。若請求失敗,`onError: handleError.bind(showErrorToast)` 會將錯誤訊息以 toast 形式顯示給使用者。成功則交由 `onSuccess` 處理導向邏輯。整個 mutation 是 `login` 這個 async function 的封裝,真正的 API 呼叫在其內部。

### 3.3 LoginService.loginAccessToken (OAuth2 表單)

`useAuth.ts` 中的 `login` 函式呼叫自動生成的 `LoginService.loginAccessToken({ formData: data })`,這是 OpenAPI codegen 根據後端 FastAPI 的 `/login/access-token` 端點產生的 client。底層走 `request.ts` 的 `request()`,因為 `formData` 存在,`getFormData()` 會將資料轉成 `FormData` 物件,符合 OAuth2 password grant 規範(`application/x-www-form-urlencoded` 或 `multipart/form-data`)。此請求**不需要**帶 Authorization header(因為此時還沒有 token),`getHeaders()` 中 token 解析結果為空字串,不會加上 `Authorization`。請求送出後等待後端驗證帳密。

### 3.4 後端回傳 access_token

後端驗證帳密成功後,回應 JSON 包含 `access_token`(與 `token_type`,通常為 `bearer`),對應 codegen 型別中的回應 schema。`request()` 函式透過 `getResponseBody()` 依 `Content-Type: application/json` 自動解析為物件。若帳密錯誤,後端回傳 401,會被 `catchErrorCodes()` 轉成 `ApiError` 並 reject,進而觸發 `loginMutation` 的 `onError`。只有狀態碼 200 且成功解析的回應,才會繼續往下走到 `onSuccess`。

### 3.5 存入 localStorage

`useAuth.ts` 透過 `localStorage.setItem("access_token", response.access_token)` 將 token 寫入瀏覽器的 localStorage,這是同步操作,寫入後立即生效。此後所有 API 請求都會經由 `main.tsx` 設定的 `OpenAPI.TOKEN` resolver,從 localStorage 讀出此 token 並加到 `Authorization: Bearer <token>` header。由於沒有 expiry 或 refresh token 機制,token 會一直存在直到使用者登出、401 被全域攔截清除,或手動清空瀏覽器資料。`isLoggedIn()` 也是依賴此 key 是否存在來判斷登入狀態。

### 3.6 導向 "/"

`useAuth.ts` 的 `onSuccess` 呼叫 `navigate({ to: "/" })`,使用 TanStack Router 的 SPA 導航(非整頁刷新)。導向 `/` 後會經過 `_layout.tsx` 的 `beforeLoad`,此時 `isLoggedIn()` 已為 true,不會被導回 `/login`,可正常進入主畫面。同時 `currentUser` 的 `useQuery`(因 `enabled: isLoggedIn()` 條件成立)會自動觸發,呼叫 `UsersService.readUserMe` 取得使用者資料並快取於 React Query。整個登入流程至此完成,後續頁面渲染依賴此快取的 user 資料。

## 4. 觀察與潛在問題

- **Token 不會過期重新整理**:沒有 refresh token 機制,token 過期只靠 401 觸發整頁強制跳轉,使用者體驗較粗暴(會丟失當前頁面狀態)。
- **localStorage 存 token** 有 XSS 風險(相較 httpOnly cookie),但這是 fastapi-template 的標準做法。
- `WITH_CREDENTIALS: false`、`CREDENTIALS: 'include'` 但未啟用,目前完全靠 Bearer token,沒用到 cookie-based session。

## 5. 額外補充:路由守衛(beforeLoad)

- `/login`、`/signup`、`/recover-password`、`/reset-password`:已登入則 redirect 到 `/`
- `/_layout`(含子路由)、`/layout2`:未登入則 redirect 到 `/login`
- `/_layout/admin`:額外檢查是否為 superuser

## 6. 全域 401/403 處理

`main.tsx` 中 `QueryCache`/`MutationCache` 的 `onError` 攔截,若收到 401/403 自動清除 token 並 `window.location.href = "/login"`(強制整頁刷新)。
