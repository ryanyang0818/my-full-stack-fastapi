# src/routes/ 說明

TanStack Router 的「檔案式路由」目錄：**檔名＝網址路徑**。這個目錄的結構會被 [routeTree.gen.ts](../../docs/目錄說明.md) 自動掃描、組成一棵路由樹。

## 總覽

| 項目 | 類型 | 分類 | 說明 | 對應網址 |
|---|---|---|---|---|
| __root.tsx | 檔案 | 套件架構本身 | 整個 App 的根路由，定義全域錯誤頁/404頁與 devtools | （所有頁面的外層） |
| _layout.tsx | 檔案 | 套件架構本身 | 登入後主版面（側邊欄＋header＋footer），含登入守衛 | `/`、`/items`、`/admin`、`/settings`（透過子路由） |
| _layout/ | 目錄 | 套件架構本身 | 主版面下的實際頁面內容 | 詳見 [_layout/docs/目錄說明.md](../_layout/docs/目錄說明.md) |
| login.tsx | 檔案 | 套件架構本身 | 登入頁 | `/login` |
| signup.tsx | 檔案 | 套件架構本身 | 註冊頁 | `/signup` |
| recover-password.tsx | 檔案 | 套件架構本身 | 忘記密碼（輸入 email 觸發寄信） | `/recover-password` |
| reset-password.tsx | 檔案 | 套件架構本身 | 重設密碼（用 email 連結中的 token） | `/reset-password?token=...` |

## 重點檔案：`__root.tsx` — 所有頁面共用的最外層骨架

```tsx
export const Route = createRootRoute({
  component: () => (
    <>
      <HeadContent />
      <Outlet />
      <TanStackRouterDevtools position="bottom-right" />
      <ReactQueryDevtools initialIsOpen={false} />
    </>
  ),
  notFoundComponent: () => <NotFound />,
  errorComponent: () => <ErrorComponent />,
})
```

不管使用者訪問哪個網址，最終畫面結構都是「這個元件 + `<Outlet />` 裡面塞進對應子路由的內容」。三個重點：

- `<HeadContent />`：讓子路由可以用 `head: () => ({ meta: [{ title: "..." }] })` 設定瀏覽器標題（[login.tsx](#重點檔案-logintsx--登入頁完整解析) 等頁面都有用到）。
- `notFoundComponent`：網址打錯（路由樹裡找不到對應頁面）時顯示 [NotFound](../../components/Common/) 元件，而不是白畫面。
- `errorComponent`：任何子路由的元件 render 時丟出例外，會被這裡的 [ErrorComponent](../../components/Common/) 接住，顯示友善的錯誤畫面而不是整個白屏當機。
- 右下角兩個 `*Devtools` 只在開發模式下會顯示浮動面板，方便除錯 router 狀態與 react-query cache。

## 重點檔案：`_layout.tsx` — 登入後主版面 + 登入守衛

```tsx
export const Route = createFileRoute("/_layout")({
  component: Layout,
  beforeLoad: async () => {
    if (!isLoggedIn()) {
      throw redirect({ to: "/login" })
    }
  },
})

function Layout() {
  return (
    <SidebarProvider>
      <AppSidebar />
      <SidebarInset>
        <header className="sticky top-0 z-10 flex h-16 shrink-0 items-center gap-2 border-b px-4">
          <SidebarTrigger className="-ml-1 text-muted-foreground" />
        </header>
        <main className="flex-1 p-6 md:p-8">
          <div className="mx-auto max-w-7xl">
            <Outlet />
          </div>
        </main>
        <Footer />
      </SidebarInset>
    </SidebarProvider>
  )
}
```

檔名 `_layout.tsx`（前面的下底線）在 TanStack Router 是特殊命名：**它本身不是一個有實際內容的頁面，而是「一群子路由共用的外殼」**。任何路徑放在 `_layout/` 資料夾底下（見 [_layout/docs/目錄說明.md](../_layout/docs/目錄說明.md)），渲染時都會先套上這個 `Layout` 元件，子頁面的內容會被塞進中間的 `<Outlet />`。

兩個關鍵機制：

1. **`beforeLoad` 登入守衛**：在這個版面（以及它所有子頁面）渲染**之前**，先檢查 [`isLoggedIn()`](../../hooks/docs/目錄說明.md)。沒登入就直接 `throw redirect({ to: "/login" })`，使用者完全不會看到主版面閃一下再跳轉。
2. **版面結構**：`AppSidebar`（左側選單）+ `SidebarTrigger`（收合按鈕，手機版會變成漢堡選單）+ 置中的 `<Outlet />`（最大寬度 `max-w-7xl`）+ 底部 `Footer`。所有「登入後」的頁面都長這個樣子，差別只在 `<Outlet />` 裡面換了什麼內容。

## 重點檔案：`login.tsx` — 登入頁完整解析

這個檔案是 4 個認證頁面（login / signup / recover-password / reset-password）共同模式的代表，其餘 3 個可以對照這裡理解差異。

#### 1. 路由設定：已登入的人不該看到登入頁

```tsx
export const Route = createFileRoute("/login")({
  component: Login,
  beforeLoad: async () => {
    if (isLoggedIn()) {
      throw redirect({ to: "/" })
    }
  },
  head: () => ({ meta: [{ title: "Log In - FastAPI Template" }] }),
})
```

跟 [_layout.tsx](#重點檔案-_layouttsx--登入後主版面--登入守衛) 的守衛方向**正好相反**：已經登入的人訪問 `/login`，會被導回首頁 `/`（沒道理一個已登入的人還停留在登入頁）。

#### 2. 表單驗證：zod schema

```tsx
const formSchema = z.object({
  username: z.email(),
  password: z.string().min(1, { message: "Password is required" })
    .min(8, { message: "Password must be at least 8 characters" }),
}) satisfies z.ZodType<AccessToken>
```

`satisfies z.ZodType<AccessToken>` 這行是一個型別檢查的小技巧：**確保這個 zod schema 的形狀，跟後端產生的 [`AccessToken`（即 `Body_login_login_access_token`）型別相容**——如果後端要求的欄位這裡漏寫了，TypeScript 編譯時就會報錯，而不是等到執行期才發現表單送出的資料格式不對。

#### 3. 提交：呼叫 `useAuth` 的 mutation

```tsx
const { loginMutation } = useAuth()
const form = useForm<FormData>({ resolver: zodResolver(formSchema), ... })

const onSubmit = (data: FormData) => {
  if (loginMutation.isPending) return
  loginMutation.mutate(data)
}
```

`zodResolver(formSchema)` 把上面的 zod schema 接進 `react-hook-form`：使用者打字離開欄位（`mode: "onBlur"`）時就會即時驗證，錯誤訊息透過 `<FormMessage />` 顯示在欄位下方。送出表單只是呼叫 [`useAuth().loginMutation.mutate(...)`](../../hooks/docs/目錄說明.md)——實際「打 API、存 token、導頁」的邏輯全部封裝在 `useAuth` 裡，這個檔案只負責「畫面與表單」。

#### 4. 畫面結構

外層包一層 [`<AuthLayout>`](../../components/Common/)（提供認證頁共用的置中卡片版面），裡面是 `shadcn/ui` 的 [`<Form>`](../../components/ui/) 系列元件 + [`<Input>`](../../components/ui/) / [`<PasswordInput>`](../../components/ui/) / [`<LoadingButton>`](../../components/ui/)（送出中會自動顯示 loading 圖示、避免重複點擊）。

## 其他認證頁面：與 `login.tsx` 的差異

### `signup.tsx`

- 多了 `full_name`、`confirm_password` 欄位，並用 `.refine(...)` 驗證兩次密碼是否一致。
- 送出前用解構 `const { confirm_password: _confirm_password, ...submitData } = data` **把 `confirm_password` 從送往後端的資料中拿掉**——它只是前端驗證用，後端 API 不需要這個欄位。
- 呼叫的是 [`useAuth().signUpMutation`](../../hooks/docs/目錄說明.md)。
- `beforeLoad` 同 login：已登入者導回 `/`。

### `recover-password.tsx`

- 表單只有一個 `email` 欄位。
- 不使用 `useAuth`，而是**直接**用 `useMutation` 包 `LoginService.recoverPassword({ email })`——因為「寄送重設密碼信」不是 `useAuth` 既有的功能，且這個動作不需要回傳 token 或更新登入狀態，所以就地用 `useMutation` + [`handleError`](../../docs/目錄說明.md) 處理即可。
- 成功後用 [`useCustomToast`](../../hooks/docs/目錄說明.md) 顯示「Password recovery email sent successfully」並 `form.reset()`（清空表單，但**不導頁**——使用者可能要去檢查信箱）。

### `reset-password.tsx`

- 路由多了 `validateSearch: searchSchema`：**從網址的 query string 讀取 `token`**（使用者點 email 裡的連結進來，網址會是 `/reset-password?token=xxx`）。
- `beforeLoad` 多一個檢查：`if (!search.token) throw redirect({ to: "/login" })`——沒帶 token 直接訪問這個頁面，視為無效，導回登入頁。
- 表單只填新密碼＋確認密碼；送出時用 `Route.useSearch()` 取出網址裡的 `token`，跟新密碼一起送給 `LoginService.resetPassword`。
- 成功後導向 `/login`（跟 recover-password 不同——這裡密碼真的改完了，理所當然該去登入）。
