# src/routes/_layout/ 說明

放在這個資料夾底下的路由，渲染時都會套上 [`_layout.tsx`](../../docs/目錄說明.md) 的版面外殼（側邊欄＋header＋footer，且需登入）。這四個檔案就是「登入後看到的四個主要頁面」。

## 總覽

| 項目 | 類型 | 分類 | 說明 | 對應網址 |
|---|---|---|---|---|
| index.tsx | 檔案 | 套件架構本身 | 首頁 Dashboard，顯示歡迎訊息 | `/` |
| items.tsx | 檔案 | 套件架構本身 | Items 清單與新增（CRUD） | `/items` |
| admin.tsx | 檔案 | 套件架構本身 | 使用者管理（僅 superuser） | `/admin` |
| settings.tsx | 檔案 | 套件架構本身 | 個人設定（分頁：個人資料／密碼／危險區） | `/settings` |

---

## `index.tsx` — 最簡單的頁面：Dashboard

```tsx
function Dashboard() {
  const { user: currentUser } = useAuth()
  return (
    <div>
      <h1>Hello, {currentUser?.full_name || currentUser?.email} 👋</h1>
      <p>Welcome back, nice to see you again!!!</p>
    </div>
  )
}
```

只用 [`useAuth()`](../../../hooks/docs/目錄說明.md) 拿到目前登入的使用者，顯示一句歡迎詞。是了解整個路由系統最好的起點：檔名 `index.tsx` 對應網址 `/`，且因為放在 `_layout/` 裡，畫面會自動被包進側邊欄版面。

## 重點檔案：`items.tsx` — 「列表 + 新增」頁面的標準模式

這個檔案示範了本專案所有「資料列表」頁面共用的模式（[admin.tsx](#admintsx--使用者管理跟-itemstsx-的差異) 幾乎是它的翻版）。可以拆成三層：

#### 第一層：定義「怎麼拿資料」

```tsx
function getItemsQueryOptions() {
  return {
    queryFn: () => ItemsService.readItems({ skip: 0, limit: 100 }),
    queryKey: ["items"],
  }
}
```

`ItemsService.readItems` 來自 [client/](../../../client/docs/目錄說明.md)（對應後端 `GET /api/v1/items/`）。把 `queryFn` / `queryKey` 包成一個函式，方便在多處重複使用同一組查詢設定（例如 [AddItem](../../../components/Items/) 新增成功後，要 `invalidateQueries({ queryKey: ["items"] })` 來刷新這個列表）。

#### 第二層：用 `Suspense` 處理「載入中」狀態

```tsx
function ItemsTableContent() {
  const { data: items } = useSuspenseQuery(getItemsQueryOptions())

  if (items.data.length === 0) {
    return ( /* 空狀態：放大鏡圖示 + "You don't have any items yet" */ )
  }
  return <DataTable columns={columns} data={items.data} />
}

function ItemsTable() {
  return (
    <Suspense fallback={<PendingItems />}>
      <ItemsTableContent />
    </Suspense>
  )
}
```

這裡的關鍵是 **`useSuspenseQuery`**（不是一般的 `useQuery`）：它在資料還沒回來時，會讓元件直接「掛起（suspend）」，不會回傳 `isLoading` 之類的狀態給你手動判斷。取而代之，外層的 `<Suspense fallback={<PendingItems />}>` 會在資料還沒準備好時，直接顯示 [`PendingItems`](../../../components/Pending/)（skeleton 骨架畫面）。

好處：`ItemsTableContent` 內部完全不用寫 `if (isLoading) return <Skeleton />`——程式碼可以**假設 `items` 一定有資料**，載入中的畫面切換交給 React 的 Suspense 機制統一處理。三種狀態分別對應三個元件：

| 狀態 | 顯示什麼 |
|---|---|
| 載入中 | `<PendingItems />`（骨架） |
| 載入完成但是空陣列 | 「You don't have any items yet」空狀態提示 |
| 有資料 | `<DataTable columns={columns} data={items.data} />`（[DataTable](../../../components/Common/) + [columns](../../../components/Items/) 定義表格欄位） |

#### 第三層：頁面骨架

```tsx
function Items() {
  return (
    <div className="flex flex-col gap-6">
      <div className="flex items-center justify-between">
        <div>
          <h1>Items</h1>
          <p>Create and manage your items</p>
        </div>
        <AddItem />
      </div>
      <ItemsTable />
    </div>
  )
}
```

標題 + [`<AddItem />`](../../../components/Items/)（右上角「新增」按鈕，會開啟 Dialog）+ 下方表格。

## `admin.tsx` — 使用者管理，跟 `items.tsx` 的差異

整體結構幾乎一致（`useSuspenseQuery` + `Suspense` + [`DataTable`](../../../components/Common/)），差異點：

1. **多一層權限守衛**：

   ```tsx
   beforeLoad: async () => {
     const user = await UsersService.readUserMe()
     if (!user.is_superuser) {
       throw redirect({ to: "/" })
     }
   }
   ```

   注意這裡**直接呼叫 API**（`UsersService.readUserMe()`），而不是用 [`useAuth()`](../../../hooks/docs/目錄說明.md) 裡 cache 住的 `user`——因為 `beforeLoad` 跑在 React 元件 render 之前，那時候 react-query 的 cache 可能還沒有資料，所以必須自己重新打一次 API 來確認「現在登入的人是不是 superuser」。不是 superuser 就直接導回首頁 `/`，**整個 `/admin` 頁面內容完全不會被渲染**。

2. **資料多了一個欄位**：

   ```tsx
   const tableData: UserTableData[] = users.data.map((user: UserPublic) => ({
     ...user,
     isCurrentUser: currentUser?.id === user.id,
   }))
   ```

   把目前登入的 `currentUser` 跟列表裡每一筆 `user` 比對，標記出「哪一筆是自己」。[Admin/columns.tsx](../../../components/Admin/) 會用這個 `isCurrentUser` 欄位決定要不要顯示「刪除自己」之類的操作（通常會擋掉）。

## `settings.tsx` — 用 Tabs 切換多個設定子頁面

```tsx
const tabsConfig = [
  { value: "my-profile", title: "My profile", component: UserInformation },
  { value: "password", title: "Password", component: ChangePassword },
  { value: "danger-zone", title: "Danger zone", component: DeleteAccount },
]

function UserSettings() {
  const { user: currentUser } = useAuth()
  const finalTabs = currentUser?.is_superuser
    ? tabsConfig.slice(0, 3)
    : tabsConfig

  if (!currentUser) return null

  return (
    <Tabs defaultValue="my-profile">
      <TabsList>
        {finalTabs.map((tab) => (
          <TabsTrigger key={tab.value} value={tab.value}>{tab.title}</TabsTrigger>
        ))}
      </TabsList>
      {finalTabs.map((tab) => (
        <TabsContent key={tab.value} value={tab.value}>
          <tab.component />
        </TabsContent>
      ))}
    </Tabs>
  )
}
```

設計重點：**用一個設定陣列 `tabsConfig` 驅動畫面**，而不是把三組 `<TabsTrigger>` / `<TabsContent>` 寫死三次——`map` 兩次就把分頁標籤與分頁內容都產生出來，未來要加第四個分頁，只要在 `tabsConfig` 多加一筆。三個分頁對應三個元件，細節見 [UserSettings/](../../../components/UserSettings/docs/目錄說明.md)：

- **My profile** → [`UserInformation`](../../../components/UserSettings/)：顯示/編輯姓名、email
- **Password** → [`ChangePassword`](../../../components/UserSettings/)：修改密碼
- **Danger zone** → [`DeleteAccount`](../../../components/UserSettings/)：刪除帳號

`if (!currentUser) return null`：在 [`useAuth()`](../../../hooks/docs/目錄說明.md) 的使用者資料還沒載入完成前，先什麼都不顯示（因為下面馬上要用到 `currentUser.is_superuser`），避免短暫出現 `undefined` 造成的錯誤畫面。

> 註：`tabsConfig` 目前正好只有 3 個項目，所以 `tabsConfig.slice(0, 3)` 與 `tabsConfig` 在結果上是一樣的（兩個分支目前等價）。這行 `is_superuser` 判斷保留了「未來如果新增第 4 個只給一般使用者看的分頁」時的擴充空間。
