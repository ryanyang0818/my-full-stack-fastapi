# src/components/ui/ 說明

[shadcn/ui](https://ui.shadcn.com/) 風格的 UI primitives：每個檔案對應一個基礎元件，內部用 [Radix UI](https://www.radix-ui.com/) 處理無障礙互動邏輯，外觀用 Tailwind + [`cn()`](../../../lib/docs/目錄說明.md) 控制。這些檔案是用 `shadcn` CLI 安裝產生的樣板，一般不會手動修改內部邏輯，只會調整樣式或新增 variant。

## 重點檔案：`sidebar.tsx` — 整套側邊欄系統

這是這個資料夾最大、也是 [AppSidebar](../../Sidebar/docs/目錄說明.md)（左側主選單）的地基。可以分成三層理解：

#### 1. `SidebarProvider` + `useSidebar()` — 「展開/收合」狀態的單一來源

```tsx
const SidebarContext = React.createContext<SidebarContextProps | null>(null)

function useSidebar() {
  const context = React.useContext(SidebarContext)
  if (!context) {
    throw new Error("useSidebar must be used within a SidebarProvider.")
  }
  return context
}
```

[`_layout.tsx`](../../../routes/docs/目錄說明.md) 整個版面最外層包了 `<SidebarProvider>`，裡面所有元件（`Sidebar`、`SidebarTrigger`、`SidebarInset`...）都透過 `useSidebar()` 拿到同一份「現在側邊欄是展開還是收合」的狀態，不需要一層層往下傳 props。

值得注意的兩個細節：

- **狀態存進 cookie**：

  ```ts
  document.cookie = `${SIDEBAR_COOKIE_NAME}=${openState}; path=/; max-age=${SIDEBAR_COOKIE_MAX_AGE}`
  ```

  使用者收合側邊欄後重新整理頁面，狀態會記住（存 cookie 而非 `localStorage`，是因為 SSR 場景下伺服器渲染時也能讀到——雖然這個專案是純前端 SPA，但這是 shadcn 樣板沿用 Next.js 場景的寫法）。

- **快捷鍵 `Ctrl/Cmd + B`**：

  ```ts
  if (event.key === SIDEBAR_KEYBOARD_SHORTCUT && (event.metaKey || event.ctrlKey)) {
    event.preventDefault()
    toggleSidebar()
  }
  ```

  跟 VS Code 的側邊欄快捷鍵習慣一致。

#### 2. `Sidebar` 元件 — 同一份內容，三種呈現方式

```tsx
function Sidebar({ side = "left", variant = "sidebar", collapsible = "offcanvas", ... }) {
  const { isMobile, state, openMobile, setOpenMobile } = useSidebar()

  if (collapsible === "none") { /* 固定顯示，不能收合 */ }
  if (isMobile) {
    return <Sheet open={openMobile} onOpenChange={setOpenMobile}>...</Sheet>  // 手機：滑出抽屜
  }
  return ( /* 桌面：固定欄位，用 data-state="expanded"/"collapsed" 控制寬度 */ )
}
```

同一個 `<Sidebar>`，根據 [`useIsMobile()`](../../../hooks/docs/目錄說明.md) 的結果，**手機上會變成 [Sheet](#其他重點sheet--dialog) 滑出抽屜**，桌面上則是用 `data-state` 屬性 + Tailwind 的 `group-data-[...]` 選擇器控制寬度在 `16rem`（展開）與 `3rem`（只剩 icon）之間切換——切換時靠 CSS transition 做動畫，不是 JS 重新 render。

#### 3. `SidebarMenu*` 系列 — 選單項目的building blocks

`SidebarMenu` / `SidebarMenuItem` / `SidebarMenuButton` / `SidebarMenuSub` 這些是 [AppSidebar](../../Sidebar/docs/目錄說明.md) 用來「拼」出選單樹的小積木：一個 `<SidebarMenu>` 是 `<ul>`，裡面每個 `<SidebarMenuItem>` 是 `<li>`，`<SidebarMenuButton>` 才是真正可點擊、可以有 icon + 文字、收合時自動只顯示 icon 並出現 tooltip 的按鈕。`SidebarMenuSkeleton` 則是選單資料載入中時的骨架佔位。

## 重點檔案：`form.tsx` — 把 react-hook-form 接到 shadcn 元件上

這個檔案**不是**一個「看得到的元件」，而是一組「Context 轉接器」，讓 [routes/login.tsx](../../../routes/docs/目錄說明.md) 這類表單可以這樣寫：

```tsx
<Form {...form}>
  <FormField
    control={form.control}
    name="username"
    render={({ field }) => (
      <FormItem>
        <FormLabel>Email</FormLabel>
        <FormControl>
          <Input {...field} />
        </FormControl>
        <FormMessage />
      </FormItem>
    )}
  />
</Form>
```

背後發生的事：

1. `Form = FormProvider`：直接把 react-hook-form 的 `FormProvider` 重新命名 export，讓整個表單樹都能用 `useFormContext()`。
2. `FormField`：包了 react-hook-form 的 `<Controller>`，並透過 `FormFieldContext` 把「目前是哪個欄位名稱（`name`）」往下傳。
3. `useFormField()`：**核心 hook**——同時讀取 `FormFieldContext`（欄位名稱）和 `FormItemContext`（一個用 `React.useId()` 產生的唯一 id），再呼叫 react-hook-form 的 `useFormState` 拿到這個欄位目前的驗證錯誤狀態 (`fieldState`)。回傳一包 `{ id, name, formItemId, formDescriptionId, formMessageId, error, ... }`。
4. `FormLabel` / `FormControl` / `FormMessage` 全部呼叫 `useFormField()`，自動把 `id`、`aria-describedby`、`aria-invalid` 這些無障礙屬性接好，並且：

   ```tsx
   function FormMessage({ className, ...props }) {
     const { error, formMessageId } = useFormField()
     const body = error ? String(error?.message ?? "") : props.children
     if (!body) return null
     return <p id={formMessageId} className="text-destructive text-sm">{body}</p>
   }
   ```

   **`FormMessage` 會自動顯示 zod 驗證失敗的錯誤訊息**——這就是為什麼 [login.tsx](../../../routes/docs/目錄說明.md) 裡 `formSchema` 定義的 `"Password must be at least 8 characters"` 這類訊息，打字打到一半離開欄位就會自動出現在 `<FormMessage />` 的位置，元件本身完全不用寫顯示邏輯。

**一句話總結**：`form.tsx` 是「zod 驗證錯誤」與「畫面上該顯示什麼」之間的接線板，本專案所有表單（登入、註冊、新增 Item/User、修改密碼...）都靠它顯示欄位錯誤。

## 其他重點：`sheet.tsx` / `dialog.tsx`

兩者都是 Radix 的「浮層」元件（`Sheet` 從邊緣滑入、`Dialog` 在畫面中央彈出），API 形狀幾乎一樣（`<Sheet>` / `<SheetContent>` / `<SheetHeader>`... 對應 `<Dialog>` / `<DialogContent>` / `<DialogHeader>`...）。專案中的用途：

- `Dialog`：[AddItem](../../Items/docs/目錄說明.md)、[AddUser](../../Admin/docs/目錄說明.md)、[EditItem](../../Items/docs/目錄說明.md) 等「新增/編輯」表單彈窗。
- `Sheet`：[sidebar.tsx](#重點檔案-sidebartsx--整套側邊欄系統) 內部用它實現手機版的側邊欄抽屜。

## `password-input.tsx` / `loading-button.tsx` — 比標準 shadcn 多一點邏輯的兩個元件

這兩個不是 shadcn 預設清單裡的元件（標準只有 `input.tsx` / `button.tsx`），是專案在 `input` / `button` 基礎上**多包一層常用行為**：

- **`PasswordInput`**：在 `<input type={showPassword ? "text" : "password"}>` 右側疊加一個 [`Button`](#總覽) 圖示（眼睛 `Eye` / `EyeOff`），點擊切換 `showPassword` 狀態——這就是登入/註冊頁「顯示密碼」按鈕的完整實作。
- **`LoadingButton`**：在 [`button.tsx`](#總覽) 定義的 `buttonVariants` 基礎上，多了一個 `loading` prop——`loading={true}` 時自動加上轉圈圈圖示（`Loader2` + `animate-spin`）並把按鈕 `disabled`，避免使用者在 API 請求送出期間重複點擊。所有表單的送出按鈕（[login.tsx](../../../routes/docs/目錄說明.md)、[AddItem](../../Items/docs/目錄說明.md) 等）都用這個而不是 `<Button>`。

## 其餘小型 primitives

`avatar.tsx`、`badge.tsx`、`card.tsx`、`alert.tsx`、`tooltip.tsx`、`tabs.tsx`、`table.tsx`、`pagination.tsx`、`select.tsx`、`dropdown-menu.tsx`、`checkbox.tsx`、`separator.tsx`、`label.tsx`、`input.tsx`、`skeleton.tsx`、`button-group.tsx`、`sonner.tsx` 都是 shadcn 標準元件，行為跟官方文件一致：包裝對應的 Radix primitive（或純 HTML 標籤），提供 `cn()` 可覆寫的預設樣式與 `data-slot` 屬性方便客製化。各自的使用範例可在對應的業務元件（[Common/](../../Common/docs/目錄說明.md)、[Items/](../../Items/docs/目錄說明.md)、[Admin/](../../Admin/docs/目錄說明.md) 等）中看到。
