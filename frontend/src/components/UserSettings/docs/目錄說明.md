## 重點檔案：`UserInformation.tsx` — 「檢視模式 / 編輯模式」切換的表單

這個檔案示範了一種很實用的表單模式：**同一個 `<FormField>`，依 `editMode` 狀態渲染成「純文字」或「輸入框」兩種樣子**。

```tsx
const [editMode, setEditMode] = useState(false)

<FormField
  control={form.control}
  name="full_name"
  render={({ field }) =>
    editMode ? (
      <FormItem>
        <FormLabel>Full name</FormLabel>
        <FormControl>
          <Input type="text" {...field} />
        </FormControl>
        <FormMessage />
      </FormItem>
    ) : (
      <FormItem>
        <FormLabel>Full name</FormLabel>
        <p className={cn("py-2 truncate max-w-sm", !field.value && "text-muted-foreground")}>
          {field.value || "N/A"}
        </p>
      </FormItem>
    )
  }
/>
```

**故事**：使用者一進到「My profile」分頁，看到的是兩行純文字（目前的姓名、Email），畫面乾淨、不像表單。點擊「Edit」按鈕後，`editMode` 變成 `true`，這兩行文字**原地**變成可編輯的 [`<Input>`](../../ui/docs/目錄說明.md)（[`form.tsx`](../../ui/docs/目錄說明.md) 的 `FormField` 提供的 `field` 物件兩種情況下都有效，因為底層的 `react-hook-form` 狀態從一開始就存在，只是 UI 呈現不同）。

修改完按「Save」，`onSubmit` 只會把**真的改過的欄位**送出去：

```ts
const onSubmit = (data: FormData) => {
  const updateData: UserUpdateMe = {}
  if (data.full_name !== currentUser?.full_name) updateData.full_name = data.full_name
  if (data.email !== currentUser?.email) updateData.email = data.email
  mutation.mutate(updateData)
}
```

例如使用者只改了 Email、姓名沒動，送往後端的 `updateData` 就只有 `{ email: "..." }`，不會多帶一個沒變化的 `full_name`。

**Save 按鈕還有一個小細節**：

```tsx
<LoadingButton type="submit" loading={mutation.isPending} disabled={!form.formState.isDirty}>
  Save
</LoadingButton>
```

`form.formState.isDirty`（react-hook-form 內建狀態：表單值是否與初始值不同）為 `false` 時，Save 按鈕直接 `disabled`——使用者點了「Edit」但什麼都沒改，就無法誤觸送出一個空白更新。「Cancel」則呼叫 `form.reset()` 把欄位還原成 `currentUser` 原本的值，再切回檢視模式。

**一句話總結**：用一個 `editMode` boolean + `form.formState.isDirty`，就做出「預設只看、點了才能改、沒改不能存」的個人資料表單，不需要額外的「暫存資料」或「取消時還原資料」邏輯——全部交給 react-hook-form 內建的表單狀態管理。

## 其他檔案說明

### `ChangePassword.tsx` — 修改密碼

三個密碼欄位（目前密碼、新密碼、確認新密碼），都用 [`<PasswordInput>`](../../ui/docs/目錄說明.md)（含顯示/隱藏切換）。zod schema 用 `.refine()` 確認「新密碼」與「確認新密碼」一致；`mode: "onSubmit"`（跟大多數表單的 `onBlur` 不同——密碼欄位通常不希望使用者打字時就跳錯誤訊息，等按下送出才驗證）。成功後呼叫 `UsersService.updatePasswordMe`，並 `form.reset()` 清空三個欄位。

### `DeleteAccount.tsx` — 危險區塊的外框

只是一個帶紅色邊框（`border-destructive/50`）的容器，標題「Delete Account」+ 說明文字 + `<DeleteConfirmation />`。本身沒有任何邏輯，純粹是排版用的外殼。

### `DeleteConfirmation.tsx` — 真正執行刪除的二次確認

點擊「Delete Account」紅色按鈕，開啟 [`Dialog`](../../ui/docs/目錄說明.md) 要求再次確認（「此操作無法復原」）。確認後呼叫 `UsersService.deleteUserMe()`，成功後呼叫 [`useAuth().logout()`](../../../hooks/docs/目錄說明.md)——帳號刪除後，使用者的 token 已經失效，必須立即登出並導回登入頁，不能繼續停留在已刪除帳號的頁面上。
