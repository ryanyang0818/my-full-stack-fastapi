# src/components/Items/ 說明

[items.tsx](../../../routes/_layout/docs/目錄說明.md)（`/items`）的 CRUD 功能：新增、編輯、刪除項目，以及表格欄位定義。結構與 [Admin/](../../Admin/docs/目錄說明.md) 完全對應（同一套「Dialog + react-hook-form + zod + useMutation」模式），只是資料變成 `Item`（`title` / `description`）而不是 `User`。

## 總覽

| 項目 | 類型 | 分類 | 說明 | 對應 Admin/ |
|---|---|---|---|---|
| columns.tsx | 檔案 | 套件架構本身 | [DataTable](../../Common/docs/目錄說明.md) 欄位定義（ID/Title/Description/操作） | [columns.tsx](../../Admin/docs/目錄說明.md) |
| AddItem.tsx | 檔案 | 套件架構本身 | 「新增 Item」按鈕 + Dialog 表單 | [AddUser.tsx](../../Admin/docs/目錄說明.md) |
| EditItem.tsx | 檔案 | 套件架構本身 | 「編輯 Item」選單項目 + Dialog 表單 | [EditUser.tsx](../../Admin/docs/目錄說明.md) |
| DeleteItem.tsx | 檔案 | 套件架構本身 | 「刪除 Item」選單項目 + 確認 Dialog | [DeleteUser.tsx](../../Admin/docs/目錄說明.md) |
| ItemActionsMenu.tsx | 檔案 | 套件架構本身 | 表格每一列右側的「...」操作選單 | [UserActionsMenu.tsx](../../Admin/docs/目錄說明.md) |

## `AddItem.tsx` / `EditItem.tsx` / `DeleteItem.tsx` / `ItemActionsMenu.tsx`

這四個檔案跟 [Admin/](../../Admin/docs/目錄說明.md) 對應檔案的程式碼結構幾乎相同，差異只在：

- 表單欄位變成 `title`（必填）+ `description`（選填），用 `ItemsService.createItem` / `updateItem` / `deleteItem`（[client/](../../../client/docs/目錄說明.md)）。
- zod schema 簡單很多（沒有密碼、沒有 `.refine()`）：

  ```ts
  const formSchema = z.object({
    title: z.string().min(1, { message: "Title is required" }),
    description: z.string().optional(),
  })
  ```

- `ItemActionsMenu.tsx` 沒有 [Admin 的「不能對自己操作」檢查](../../Admin/docs/目錄說明.md)——任何 Item 都可以被其擁有者編輯/刪除，不需要排除「自己」這種特例。
- 成功後都呼叫 `queryClient.invalidateQueries({ queryKey: ["items"] })`，讓 [items.tsx](../../../routes/_layout/docs/目錄說明.md) 的列表重新抓取最新資料。

想了解這套「Dialog 表單 + Mutation」模式的完整解析，請見 [Admin/docs/目錄說明.md](../../Admin/docs/目錄說明.md#重點檔案edituser tsx--密碼選填的編輯表單) 的 `EditUser.tsx`（兩者邏輯一致，只是欄位不同）。

## 重點檔案：`columns.tsx` — 多了一個「複製 ID」的小功能

跟 [Admin/columns.tsx](../../Admin/docs/目錄說明.md) 同樣是 `ColumnDef<ItemPublic>[]`，定義 4 欄：ID、Title、Description、Actions。值得拿出來講的是 ID 欄的 `CopyId` 元件：

```tsx
function CopyId({ id }: { id: string }) {
  const [copiedText, copy] = useCopyToClipboard()
  const isCopied = copiedText === id

  return (
    <div className="flex items-center gap-1.5 group">
      <span className="font-mono text-xs text-muted-foreground">{id}</span>
      <Button
        variant="ghost"
        size="icon"
        className="size-6 opacity-0 group-hover:opacity-100 transition-opacity"
        onClick={() => copy(id)}
      >
        {isCopied ? <Check className="size-3 text-green-500" /> : <Copy className="size-3" />}
      </Button>
    </div>
  )
}
```

實際使用情境：Item 的 `id` 是一串很長的 UUID，使用者想把它複製去呼叫 API 或回報問題。把滑鼠移到 ID 上時（`group-hover:`），右邊會浮現一個複製圖示按鈕；點擊後呼叫 [`useCopyToClipboard()`](../../../hooks/docs/目錄說明.md) 把 `id` 寫入剪貼簿，圖示也會**從 `Copy` 變成綠色的 `Check`**——因為 `copiedText === id`，這個比對成立的條件正是「`useCopyToClipboard` 內部的 `copiedText` state 等於剛剛複製的那個 `id`」。2 秒後 `useCopyToClipboard` 會自動把 `copiedText` 重設為 `null`，圖示就變回 `Copy`。

`Description` 欄：如果 `description` 是空的，顯示斜體的「No description」灰字（`!description && "italic"`，用 [`cn()`](../../../lib/docs/目錄說明.md) 動態加 class）。

`Actions` 欄：`<ItemActionsMenu item={row.original} />`，內部組合 `EditItem` + `DeleteItem`，跟 [Admin 的 UserActionsMenu](../../Admin/docs/目錄說明.md) 結構相同但沒有「排除自己」的判斷。
