## 重點檔案：`EditUser.tsx` — 「密碼選填」的編輯表單

跟新增使用者（`AddUser.tsx`）共用幾乎一樣的表單欄位（Email、姓名、密碼、確認密碼、是否 superuser、是否啟用），但編輯時**密碼欄位是選填的**——使用者編輯資料時，通常只是想改個名字，不一定要重設密碼。zod schema 因此要處理「空字串也算合法」：

```ts
const formSchema = z
  .object({
    email: z.email({ message: "Invalid email address" }),
    full_name: z.string().optional(),
    password: z
      .string()
      .min(8, { message: "Password must be at least 8 characters" })
      .optional()
      .or(z.literal("")),          // 空字串也視為合法
    confirm_password: z.string().optional(),
    is_superuser: z.boolean().optional(),
    is_active: z.boolean().optional(),
  })
  .refine((data) => !data.password || data.password === data.confirm_password, {
    message: "The passwords don't match",
    path: ["confirm_password"],
  })
```

- `.optional().or(z.literal(""))`：欄位可以「不填」或「填空字串」——但**只要有填，就至少要 8 個字**。`<input>` 沒填時 react-hook-form 預設值是 `""`，所以必須額外允許空字串通過驗證。
- `.refine((data) => !data.password || ...)`：密碼比對規則改成「如果沒填密碼，就跳過確認密碼的比對」——避免使用者不改密碼時，因為兩個欄位都是空字串卻被某些寫法誤判。

送出前的處理：

```ts
const onSubmit = (data: FormData) => {
  // exclude confirm_password from submission data and remove password if empty
  const { confirm_password: _, ...submitData } = data
  if (!submitData.password) {
    delete submitData.password
  }
  mutation.mutate(submitData)
}
```

跟 [signup.tsx](../../../routes/docs/目錄說明.md) 一樣用解構拿掉 `confirm_password`；如果密碼欄位是空的，連 `password` 這個 key 都從送往後端的物件中整個移除——後端 `UserUpdate` schema 把 `password` 視為「有傳就改、沒傳就不動」，傳空字串反而會被當成「改成空密碼」。

另一個跟 `AddUser.tsx` 不同的地方：`EditUser` 是**從 [UserActionsMenu](#useractionsmenutsx--表格列的操作選單) 的下拉選單觸發**，而不是獨立按鈕，因此它本身就是一個 `<DropdownMenuItem onSelect={(e) => e.preventDefault()}>`——`preventDefault()` 是為了阻止點擊這個選單項目時，`DropdownMenu` 自動關閉並把焦點搶走（否則接著要開的 `Dialog` 會跟 `DropdownMenu` 的關閉動畫互相干擾）。

## 其他檔案說明

### `AddUser.tsx` — 新增使用者

頁面右上角「Add User」按鈕，點擊開啟 [`Dialog`](../../ui/docs/目錄說明.md) 表單（Email、姓名、密碼、確認密碼、is_superuser/is_active 兩個 [`Checkbox`](../../ui/docs/目錄說明.md)）。新增時密碼為**必填**（跟 `EditUser` 的「選填」相反），用 `.refine()` 確保兩次密碼輸入一致。送出呼叫 `UsersService.createUser`，成功後 `queryClient.invalidateQueries({ queryKey: ["users"] })` 讓 [admin.tsx](../../../routes/_layout/docs/目錄說明.md) 的列表自動刷新。

### `DeleteUser.tsx` — 刪除使用者

`UserActionsMenu` 下拉選單裡的「Delete User」，點擊開啟確認 `Dialog`（提示「此操作無法復原，且會連同此使用者的所有 Items 一併刪除」）。確認後呼叫 `UsersService.deleteUser({ userId: id })`。

### `UserActionsMenu.tsx` — 表格列的操作選單

```tsx
export const UserActionsMenu = ({ user }: UserActionsMenuProps) => {
  const { user: currentUser } = useAuth()

  if (user.id === currentUser?.id) {
    return null
  }

  return (
    <DropdownMenu open={open} onOpenChange={setOpen}>
      <DropdownMenuTrigger asChild><Button variant="ghost" size="icon"><EllipsisVertical /></Button></DropdownMenuTrigger>
      <DropdownMenuContent align="end">
        <EditUser user={user} onSuccess={() => setOpen(false)} />
        <DeleteUser id={user.id} onSuccess={() => setOpen(false)} />
      </DropdownMenuContent>
    </DropdownMenu>
  )
}
```

**`if (user.id === currentUser?.id) return null`**：如果這一列是目前登入的自己，整個「...」選單就不顯示——避免使用者在 `/admin` 頁面把自己刪除或停用。`onSuccess={() => setOpen(false)}` 讓 `EditUser`/`DeleteUser` 內的 Dialog 成功送出後，連帶把這個外層的 `DropdownMenu` 也關閉。

### `columns.tsx` — 表格欄位定義

定義 [`Admin/columns.tsx`](#總覽) 在 [`DataTable`](../../Common/docs/目錄說明.md) 裡顯示的 5 欄：

| 欄位 | 內容 |
|---|---|
| Full Name | 姓名；若是目前登入者本人，姓名旁多顯示一個「You」[`Badge`](../../ui/docs/目錄說明.md) |
| Email | 純文字 |
| Role | `is_superuser` → `Badge` 顯示 "Superuser" 或 "User" |
| Status | 一個綠/灰色小圓點 + "Active"/"Inactive" 文字 |
| Actions | `<UserActionsMenu user={row.original} />` |

`UserTableData = UserPublic & { isCurrentUser: boolean }`——`isCurrentUser` 這個額外欄位是 [admin.tsx](../../../routes/_layout/docs/目錄說明.md) 在組資料時加上去的（比對 `currentUser.id === user.id`），這裡的 "You" Badge 就是用它來判斷。
