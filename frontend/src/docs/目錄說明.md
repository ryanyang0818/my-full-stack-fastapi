# src/ 說明

前端原始碼根目錄，包含 API client、React 元件、自訂 hooks、工具函式與 TanStack Router 路由。

## 總覽

| 項目 | 類型 | 分類 | 說明 |
|---|---|---|---|
| client/ | 目錄 | 套件架構本身 | openapi-ts 產生的 API client（schemas/sdk/types）與底層 request 邏輯 |
| components/ | 目錄 | 套件架構本身 | React 元件（UI primitives、業務元件、版面） |
| hooks/ | 目錄 | 套件架構本身 | 自訂 React hooks（認證、toast、clipboard、響應式） |
| lib/ | 目錄 | 套件架構本身 | 工具函式（`cn` className 合併等） |
| routes/ | 目錄 | 套件架構本身 | TanStack Router 檔案式路由（登入、主版面、admin/items/settings） |
| index.css | 檔案 | 套件架構本身 | Tailwind 與全域樣式 |
| main.tsx | 檔案 | 套件架構本身 | React 應用進入點 |
| routeTree.gen.ts | 檔案 | 自動生成 | TanStack Router CLI 自動產生的路由樹 |
| utils.ts | 檔案 | 套件架構本身 | 全域工具函式 |
| vite-env.d.ts | 檔案 | 套件架構本身 | Vite 環境變數型別宣告 |

目錄項目的詳細內容，請見各自資料夾下的 `docs/目錄說明.md`。下面針對「核心檔案」用故事方式說明，其餘檔案用中等長度說明。

## 重點檔案說明

### `main.tsx` — 整個 App 的開機程序

把這個檔案想成「App 開機時要做的一連串設定動作」，順序大致是：

1. **告訴 API client 去哪裡找後端**

   ```ts
   OpenAPI.BASE = import.meta.env.VITE_API_URL
   ```

   這一行就是用到 [vite-env.d.ts](#vite-envdts--告訴-typescript-env-裡多了什麼變數) 宣告的 `VITE_API_URL`，把 `.env` 裡設定的後端網址（例如 `http://localhost:8000`）餵給 API client，之後所有 `client/sdk.gen.ts` 產生的 API 呼叫都會打到這個網址。

2. **告訴 API client 怎麼拿到登入 token**

   ```ts
   OpenAPI.TOKEN = async () => {
     return localStorage.getItem("access_token") || ""
   }
   ```

   每次呼叫 API 前，client 會先呼叫這個函式，從 `localStorage` 拿出登入時存的 token，塞進 `Authorization` header。

3. **設定「API 回 401/403 時自動登出」的全域規則**

   ```ts
   const handleApiError = (error: Error) => {
     if (error instanceof ApiError && [401, 403].includes(error.status)) {
       localStorage.removeItem("access_token")
       window.location.href = "/login"
     }
   }
   ```

   這個函式被註冊到 `QueryClient` 的 `QueryCache` 和 `MutationCache` 的 `onError`。意思是：**只要任何一個 API 請求（不管是查詢還是新增/修改/刪除）回應 401 或 403**，就把 token 清掉並導回登入頁。這樣每個頁面元件就不用各自處理「token 過期怎麼辦」。

4. **建立 router，並把它的型別「註冊」給 TanStack Router**

   ```ts
   const router = createRouter({ routeTree })
   declare module "@tanstack/react-router" {
     interface Register {
       router: typeof router
     }
   }
   ```

   `routeTree` 來自 [routeTree.gen.ts](#routetreegents--路由樹自動產生檔)（自動產生的路由樹）。下面的 `declare module` 是讓 `<Link to="...">` 在打字時能自動補全所有合法路徑、並在路徑打錯時報錯。

5. **把畫面真正渲染出來**

   ```tsx
   ReactDOM.createRoot(document.getElementById("root")!).render(
     <StrictMode>
       <ThemeProvider defaultTheme="dark" storageKey="vite-ui-theme">
         <QueryClientProvider client={queryClient}>
           <RouterProvider router={router} />
           <Toaster richColors closeButton />
         </QueryClientProvider>
       </ThemeProvider>
     </StrictMode>,
   )
   ```

   由外到內依序套上：暗色模式預設的主題（`ThemeProvider`）→ react-query 的全域 cache（`QueryClientProvider`）→ 真正的路由畫面（`RouterProvider`）。`Toaster` 是右下角的提示訊息元件，全站共用。

**一句話總結**：`main.tsx` 不寫任何「畫面」，它只負責「在第一個畫面出現之前，把 API 網址、登入機制、錯誤處理、主題、路由這些全域規則都裝好」。

### `utils.ts` — 兩個到處都會用到的小工具

這個檔案只有兩個 export，但幾乎每個有表單或 API 呼叫的元件都會用到。

#### `handleError` — 把後端的錯誤訊息「翻譯」成一句話，丟給 toast

來看它怎麼被使用（典型用法在各個 `Add*.tsx` / `Edit*.tsx` 元件裡）：

```ts
const mutation = useMutation({
  // ...
  onError: (err: ApiError) => {
    handleErrorFn(err)
  },
})
```

`handleError` 的內部邏輯：

```ts
export const handleError = function (
  this: (msg: string) => void,
  err: ApiError,
) {
  const errorMessage = extractErrorMessage(err)
  this(errorMessage)
}
```

注意這裡用了 `this`——這個函式被設計成「**先綁定一個顯示訊息的函式，再呼叫**」。實際使用時會搭配 [useCustomToast](../hooks/) 這類 hook：

```ts
const { showErrorToast } = useCustomToast()
const handleErrorFn = handleError.bind(showErrorToast)
```

也就是說：

1. 後端可能回傳很多種錯誤格式（FastAPI 的驗證錯誤陣列 `[{msg: "..."}]`、單一字串 `detail`、或 axios 本身的網路錯誤）。
2. `extractErrorMessage` 負責把這些**不同格式**統一「翻譯」成一句人看得懂的字串。
3. `handleError` 再把這句話，丟給 `this`（也就是 `showErrorToast`），畫面上就會跳出一個紅色錯誤提示。

**好處**：每個用到 mutation 的元件，只要 `onError: handleErrorFn`，不用自己寫一大堆 if-else 判斷錯誤格式。

#### `getInitials` — 把名字變成大頭貼縮寫

```ts
export const getInitials = (name: string): string => {
  return name
    .split(" ")
    .slice(0, 2)
    .map((word) => word[0])
    .join("")
    .toUpperCase()
}
```

舉例：

| 輸入 | 輸出 | 說明 |
|---|---|---|
| `"Ryan Yang"` | `"RY"` | 取前兩個字的首字母 |
| `"Alice"` | `"A"` | 只有一個字就只取一個字母 |
| `"john doe smith"` | `"JD"` | 最多只取前兩個字，第三個字（smith）忽略 |

用在 [Sidebar/User.tsx](../components/Sidebar/User.tsx) 之類的地方，當使用者沒有設定頭像圖片時，用「名字縮寫」當作 Avatar 的 fallback 文字。

## 其他檔案說明

### `vite-env.d.ts` — 告訴 TypeScript「env 裡多了什麼變數」

這是一個**只在打字檢查階段有用、執行時完全不會被打包進去**的型別宣告檔。它做兩件事：

1. `/// <reference types="vite/client" />` 載入 Vite 內建的型別（讓 TS 認得 `import.meta.env`、`import.meta.hot` 等 Vite 專屬語法）。
2. 用 TypeScript 的「介面合併」幫 `ImportMetaEnv` 多加一個欄位：`VITE_API_URL: string`。

如果把這個檔案刪掉，[main.tsx](#maintsx--整個-app-的開機程序) 裡的 `import.meta.env.VITE_API_URL` 在 `bun run build` 時會報錯：

```
Property 'VITE_API_URL' does not exist on type 'ImportMetaEnv'.
```

未來如果在 `.env` 新增 `VITE_XXX` 變數並要在程式碼用 `import.meta.env.VITE_XXX`，就要回來這裡多加一行宣告。

### `index.css` — 全站的顏色與樣式變數

這是整個 App 唯一的全域 CSS 入口（在 [main.tsx](#maintsx--整個-app-的開機程序) 用 `import "./index.css"` 載入一次）。內容分三塊：

1. `@import "tailwindcss"` + `@import "tw-animate-css"` — 載入 Tailwind 本體與動畫工具類。
2. `@theme inline { ... }` — 把一批 CSS 變數（如 `--color-primary`、`--color-sidebar`）對應成 Tailwind 的 `bg-primary`、`text-sidebar-foreground` 這類 class 可以用的顏色。
3. `:root { ... }` 與 `.dark { ... }` — 分別定義「亮色模式」與「暗色模式」下，每個顏色變數實際的數值（用 `oklch()` 色彩空間）。

例如改 `:root` 裡的 `--primary: oklch(0.5982 0.10687 182.4689)`，全站所有用 `bg-primary` / `text-primary` 的按鈕、連結顏色都會一起變。要新增主題色或調整品牌色，主要就是改這個檔案。

### `routeTree.gen.ts` — 路由樹自動產生檔

由 `@tanstack/router-plugin`（Vite 開發伺服器啟動時或 `bun run build` 時）根據 `src/routes/` 底下的檔案結構自動掃描、自動產生。**禁止手動編輯**——下次跑 `bun run dev` 或 build 就會被覆蓋。

它把 `routes/__root.tsx`、`routes/_layout.tsx`、`routes/_layout/index.tsx` 這些檔案，依照命名規則（檔名 = 路徑）組成一棵巢狀的路由樹，匯出給 [main.tsx](#maintsx--整個-app-的開機程序) 的 `createRouter({ routeTree })` 使用。新增/刪除一個 `routes/` 底下的檔案，這個檔案就會自動跟著變。
