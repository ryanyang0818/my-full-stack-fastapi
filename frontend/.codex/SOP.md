# SOP

此 SOP 用於使用者輸入階段關鍵字時，依序協作。
若使用者指定單一階段，只執行該階段。
若專案 `AGENTS.md` 衝突，以 `AGENTS.md` 優先。

## 1. 探索、提出計畫

- 先確認探索範圍。
- 使用 `rg` 搜尋相關元件、hook、type、style。
- 不直接假設專案結構。
- 列出預計影響檔案數量與路徑。
- 輸出 `<proposed_plan>`。
- 完成後詢問是否要「存入計畫」或「直接實作」。

## 2. 存入計畫

- 僅在使用者要求保存計畫時執行。
- 存入 `docs/執行計畫/`。
- 檔名使用 `YYYY-MM-DD-短標題.md`。
- 計畫不包含 `npm run build`。
- 此階段不測試、不設計 Playwright 情境。
- 完成後詢問是否實作。

## 3. 實作

- 使用 `apply_patch` 修改檔案。
- 保持修改範圍小。
- 優先沿用既有設計系統與元件。
- 每個 function 上方保留簡短中文註解。
- 使用 2-space indentation。
- 避免 premature abstraction。
- 中間有疑問先提出，除非使用者使用「追求目標」或 `/goal`。
- 此階段不測試、不設計 Playwright 情境。

## 4. 測試

- 先詢問使用者要如何驗證。
- 一般修改可先用人工檢查、diff、局部指令驗證。
- UI 互動改動再討論是否設計 Playwright 測試。
- 若使用者要 commit，commit 前必須跑測試。
- 回報修改差異與驗證結果。

## 5. 摘要

- 簡短列出改了什麼。
- 說明驗證結果。
- 若有未完成或無法驗證，明確說原因。
- 若使用者要求 commit，先詢問 commit message。
- 若使用者要求計畫歸檔，移入 `docs/執行計畫/完成計畫/`。
- 提醒清理上下文

## 還原規則

- 預設只還原 Codex 本輪造成的變更。
- 不還原使用者既有修改。
- 若使用者明確指定路徑或要求全部還原，先列出將被還原的檔案再執行。
