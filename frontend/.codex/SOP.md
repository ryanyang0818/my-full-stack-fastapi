# SOP 記錄使用者打下"關鍵字"的時候，詢問相關動作是否執行

## 探索

- 詢問使用者探索範圍
- 使用 `rg` 搜尋相關元件、hook、type、style。
- 不直接假設專案結構。

## 計畫

- 重要任務先提出計畫。
- 若使用者要求保存計畫，存入 `docs/執行計畫/`。
- 計畫中不要把 `npm run build` 放進測試項目。

## 實作

- 使用 `apply_patch` 修改檔案。
- 保持修改範圍小。
- 優先沿用既有設計系統與元件。
- 每個 function 上方保留簡短中文註解。
- 使用 2-space indentation。
- 避免 premature abstraction。

## 驗證

- 詢問使用者如何驗證
- 若使用者要 commit，commit 前必須跑Playwright 測試。
- 給使用者判斷，並列出修改的程式碼區別

## git；歸檔；保存計畫

- 詢問是否 commit comment
- 若使用者要求保存計畫，存入 `docs/執行計畫/完成計畫`。


## 6. 回報

- 簡短列出改了什麼。
- 說明驗證結果。
- 若有未完成或無法驗證的部分，明確說出原因。
